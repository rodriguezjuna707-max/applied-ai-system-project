CORE AI FEATURE: RAG + Claude API  [earns the 3 required points]
-----------------------------------------------------------------
The existing score_song() / recommend_songs() IS the retrieval layer.
Claude API is the generation layer -- receives scored songs as context
and generates a personalized natural language explanation.
This is true RAG: retrieve -> augment -> generate.

STRETCH FEATURE A: Multi-Source RAG  [+2pts]
---------------------------------------------
Add a second retrieval source: data/genre_descriptions.txt
  - A plain-text file with 1-2 paragraph descriptions of each genre
    (e.g., "Lofi hip hop is characterized by...").
  - When building the Claude context block, also retrieve the genre
    description matching the user's preferred genre and inject it.
  - Claude then reasons over BOTH the scored songs AND genre knowledge.
  - Function: load_genre_context(genre: str) -> str
    reads genre_descriptions.txt and returns the matching paragraph.
  - Impact on output: Claude can explain why a genre fits the user's
    emotional state, not just that the genre label matched.

STRETCH FEATURE B: Agentic Workflow  [+2pts]
---------------------------------------------
New file: src/agent.py
Function: run_agent(user_query: str, songs: list) -> dict

Multi-step chain with observable intermediate output:
  Step 1 -- PARSE: Claude reads the user's free-text query and extracts
    structured preferences as JSON:
    {"genre": "lofi", "mood": "chill", "energy": 0.3, ...}
    Log: "AGENT STEP 1 -- Parsed preferences: {...}"

  Step 2 -- RETRIEVE: Pass extracted prefs to recommend_songs().
    Log: "AGENT STEP 2 -- Retrieved 5 candidates: [titles]"

  Step 3 -- EVALUATE: Claude receives the top-5 results and scores the
    overall match quality on a 1-10 scale with a one-sentence rationale.
    Log: "AGENT STEP 3 -- Match quality: 8/10 -- reason: ..."

  Step 4 -- REFINE (conditional): If match quality < 6, Claude adjusts
    one preference (e.g., loosens genre constraint) and re-runs Step 2.
    Log: "AGENT STEP 4 -- Refining: relaxed genre constraint -> retrying"
    Maximum 1 retry to avoid infinite loops.

  Step 5 -- EXPLAIN: Call explain_with_claude() on final results.

  Return: {"steps": [list of step logs], "recommendations": top_songs,
           "explanation": str, "match_quality": int}

  The Streamlit UI exposes an "Advanced Mode" toggle that shows the
  agent's step logs in an st.expander("Agent reasoning steps").
  This makes intermediate steps observable in output (required for credit).

GUARDRAILS (required 3pts):
----------------------------
These must all be demonstrated with printed examples in the README:

  1. Input validation in agent.py:
     - If Claude returns malformed JSON in Step 1, fall back to default
       preferences and log WARNING: "Could not parse query, using defaults"
     - Example to show: input "asdfghjkl" -> graceful fallback

  2. Output guardrail in rag_explainer.py:
     - If Claude explanation is < 20 chars -> use rule-based fallback
     - Log WARNING: "Claude response too short, using fallback"

  3. API error guardrail:
     - Wrap all API calls in try/except anthropic.APIError
     - Return fallback, never crash
     - Log ERROR with exception details

  4. CSV guardrail in load_songs():
     - File not found -> raise FileNotFoundError("songs.csv not found at {path}")
     - Missing required column -> raise ValueError("Missing column: {col}")

STRETCH FEATURE C: Test Harness / Evaluation Script  [+2pts]
--------------------------------------------------------------
New file: eval.py (project root, runs standalone)

Evaluates the system on 8 predefined inputs and prints a summary:

  INPUTS (define as a list of dicts at top of file):
    1. HIGH_ENERGY_POP profile (genre=pop, mood=happy, energy=0.9)
       Expected: top result genre == "pop" OR mood == "happy"
    2. CHILL_LOFI profile
       Expected: top result energy < 0.5
    3. DEEP_INTENSE_ROCK profile
       Expected: top result genre == "rock" OR mood == "intense"
    4. RARE_GENRE (bossa nova) -- adversarial
       Expected: score < 4.0 (no perfect match exists -- system should not
       hallucinate; it should return best available with lower score)
    5. PERFECTLY_AVERAGE (all 0.5) -- adversarial
       Expected: returns 5 results (system never crashes on degenerate input)
    6. Empty query string -- guardrail test
       Expected: returns fallback message, does not crash
    7. Query with 1000 character gibberish -- guardrail test
       Expected: returns fallback or default prefs, does not crash
    8. ACOUSTIC_METALHEAD (contradictory prefs) -- adversarial
       Expected: returns results (system handles impossible combos gracefully)

  FOR EACH INPUT: run recommend_songs(), check assertion, print PASS/FAIL
  At end: print summary table:
    "Passed: 7/8  |  Failed: 1/8  |  Pass rate: 87.5%"
    List any failures with the actual vs expected value.

  eval.py does NOT call the Claude API (tests the retrieval layer only,
  so it runs in any environment without a key).

RUBRIC ITEM 7 (3pts) -- Reflection on AI Collaboration
===========================================================================
Update reflection.md -- add a new section titled:
"AI Collaboration Reflection"

Must address all 3 bullet points to earn full 3 points:

1. How AI was used during development:
   - Used Claude to generate the scoring weight system and POINTS dict
   - Used Claude to design the RAG prompt structure
   - Used Claude to debug the UserProfile->dict key mismatch
     (favorite_genre vs genre key expected by score_song)
   - Used Claude to write boilerplate (dataclass defaults, logging config)

2. One helpful AI suggestion + one flawed suggestion:
   HELPFUL: Claude suggested the sqrt curve for popularity scoring
   (pts * sqrt(pop/100)) which correctly compresses the gap between 80
   and 100 so moderately popular songs still earn ~80% of the reward.
   This was non-obvious and meaningfully improved recommendation quality.

   FLAWED: Claude initially suggested using st.chat_input() for the UI,
   which would require managing a full conversation history loop. This was
   unnecessary complexity for a retrieval-then-display pattern. The
   sidebar + results panel is simpler and makes the RAG flow more visible.

3. System limitations and future improvements:
   LIMITATIONS:
   - Catalog is only 18 songs -- scores are meaningful but variety is low
   - Genre matching is exact string match -- "hip hop" != "hip-hop"
   - Agent retry logic only fires once -- a real agent might loop 3-5 times
   - No user account system -- preferences reset on every page load

   FUTURE IMPROVEMENTS:
   - Replace CSV with a vector database (e.g., ChromaDB) for semantic search
   - Add Spotify API as a second retrieval source (real catalog, millions of songs)
   - Fine-tune Claude with music domain examples for richer explanations
   - Add user feedback loop: thumbs up/down to improve future recommendations
